# 一人公司 Agent 体系评估报告

> 评估时间：2026-06-13 | 评估视角：CEO + 行业对标
> 评估对象：9 角色 + 25 技能 + 产品闭环流程

---

## 总体评价

**结论：框架设计在国内一人公司/独立开发者圈子里属于上游水平，但距离"行业领先"有明显差距。**

具体打分：

| 维度 | 评分 | 说明 |
|------|------|------|
| 角色体系完整性 | 7/10 | 9 角色覆盖了主要职能，但缺少关键角色 |
| 协作流程设计 | 6/10 | 有闭环但偏理想化，落地细节不够 |
| 技能实操性 | 7/10 | 时间盒+反模式是亮点，但内容偏模板 |
| 差异化竞争力 | 5/10 | 和市面上的 Agent 框架相比，没有技术壁垒 |
| 极速交付能力 | 6/10 | 目标设得漂亮，但没验证过能否做到 |
| 数据驱动能力 | 6/10 | 有框架但缺工具链落地 |

## 真正的亮点（领先点）

### 1. 角色分离 + 对抗机制

```
@dev 写代码 → @qa 找问题（对抗视角）
@pm 定需求 → @data 用数据质疑需求（对抗视角）
```

大多数团队（甚至是中大型公司）的开发和测试是同一拨人，**确认偏差**导致自己写的 bug 自己看不到。一人公司用 Agent 角色分离来弥补这个天然缺陷——这是**真正的创新点**。

### 2. 时间盒驱动 + 反模式清单

每个 Skill 都有明确的时间盒和反模式清单，这比大多数公司的流程文档都要实用。大公司的流程文档没人看，一人公司的流程是**真的在用**。

### 3. 3 天出 Demo 的节奏设计

大多数初创公司还在"先做 3 个月再上线"的思维里。3 天出 Demo + 数据验证的模式，速度上碾压传统打法。

## 明显的短板（差距点）

### 1. 缺少技术架构师角色 — 最大短板

现在的 `@dev` 既是架构师又是码农，但 AI Agent 产品的**架构决策**和**代码实现**是完全不同的能力维度：

| | 架构师 | 码农 |
|--|--------|------|
| 思考方式 | 系统、全局、长期 | 功能、局部、当前 |
| 产出 | 架构图、技术选型文档、接口规范 | 代码、测试、部署 |
| 失败代价 | 选错技术栈 = 重写 | 写错代码 = 改 |

> 大公司有 CTO/架构师角色做技术方向把控，一人公司最容易在技术选型上翻车——3 个月后发现选错框架，一切推倒重来。

### 2. Agent 之间没有真正的"协作"机制

现在的 WORKFLOW.md 是**串行流水线**，不是真正协作：

```
现状：@pm → @designer → @dev → @qa → @devops（单向传递）
领先：Agent 之间能主动发起对话、质疑、协作
```

大公司的高级团队特征是：**成员之间会主动挑战彼此**。PM 挑战 Dev 的技术方案，Dev 挑战 PM 的需求合理性。当前设计缺少这个维度。

### 3. 没有记忆和学习机制

所有 Agent 和 Skill 都是**静态的**——每次调用从零开始。行业领先水平是：

| 级别 | 能力 | 当前状态 |
|------|------|----------|
| L1 静态 | 每次从零开始 | ← 我们在这里 |
| L2 项目记忆 | 记住项目上下文、技术栈、历史决策 | ❌ 缺失 |
| L3 组织学习 | 从历史项目中提炼模式，新项目自动套用 | ❌ 缺失 |
| L4 自进化 | Agent 根据效果数据自动优化自己的行为 | ❌ 缺失 |

### 4. 缺少产品经理的"用户验证"闭环

`@pm` 有 `pm-user-feedback-loop` 技能，但整个体系中**没有定义如何触达真实用户**：

| 环节 | 有 | 缺 |
|------|----|----|
| 定义需求 | ✅ @pm | |
| 设计原型 | ✅ @designer | |
| 开发产品 | ✅ @dev | |
| 部署上线 | ✅ @devops | |
| **触达用户** | ⚠️ @ops 只管内容 | ❌ 没有用户招募渠道 |
| **收集反馈** | ⚠️ @data 只看数据 | ❌ 没有定性反馈机制 |
| **验证假设** | ⚠️ @data 出报告 | ❌ 谁来决策"假设成立"？没有明确 |

### 5. 和行业领先水平的直接对比

| 维度 | 我们 | 行业领先 | 差距 |
|------|------|----------|------|
| Agent 角色数 | 9 | CrewAI/MetaGPT 有无限扩展 | 数量不是问题，深度才是 |
| Agent 通信 | 无（单向流水线） | CrewAI 支持委托/协作/辩论 | 🔴 差距大 |
| Agent 记忆 | 无 | AutoGen/MemGPT 有长期记忆 | 🔴 差距大 |
| 工具调用 | 静态定义 | LangGraph 支持动态工具 | 🟡 中等差距 |
| 流程编排 | 手动 WORKFLOW.md | LangGraph/Dify 可视化编排 | 🟡 中等差距 |
| 评估体系 | 无 | LangSmith 有 eval 框架 | 🔴 差距大 |

## 要达到"领先水平"需要补什么

### 优先级排序（影响大 → 小）

| 优先级 | 补什么 | 为什么 |
|--------|--------|--------|
| **P0** | Agent 协作机制：委托、质疑、辩论 | 没有协作的 Agent 只是角色扮演，不是真团队 |
| **P0** | 项目记忆系统：记住技术栈、架构决策、历史教训 | 否则每次从零开始，永远在重复犯错 |
| **P1** | 技术架构师角色 | AI Agent 产品的架构选型错误代价巨大 |
| **P1** | 用户触达+定性反馈机制 | 只有数据没有用户原话 = 知道数字不知道原因 |
| **P2** | 效果评估体系 | Agent 的输出好不好？需要可量化的 eval |
| **P2** | 动态工具调用 | Agent 能根据任务自行选择工具 |
| **P3** | 可视化流程编排 | 用图形界面编排 Agent 流程 |

---

## 一句总结

> **当前设计是一人公司的"管理框架"，不是"执行引擎"。**
>
> 它告诉你**该做什么角色、按什么流程走**，但每个角色真正执行时，还是靠 Claude 的通用能力。
>
> 行业领先水平是：Agent 不只是角色卡，而是有**记忆、协作能力、评估反馈**的自治系统。
>
> **下一步最值得投入的：给 Agent 加上协作机制和项目记忆——从"角色扮演"变成"真团队"。**
