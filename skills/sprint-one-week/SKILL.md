# 一周一迭代

一人公司 0→1 阶段的节奏：每周出一个可验证的成果。不是每周写一堆代码，是每周验证一个假设。

## 准备

- **季度 OKR**：本周要推进哪个 KR
- **上周复盘结论**：什么有效、什么无效
- **当前最大瓶颈**：本周最该推进什么

## 周一：规划（20min）

### Step 1：本周目标（5min）

从 OKR 的 KR 中提取本周要推进的一个目标。

> 一周只做一件事。做了 5 件事但没一件做完 = 浪费一周。

格式：
```
本周目标：验证 [假设X]
成功标准：[具体数字]
时间预算：[小时数]
```

### Step 2：任务拆解（10min）

把目标拆成 ≤ 5 个可交付的任务：

| 任务 | 负责角色 | 交付物 | 预计时间 |
|------|----------|--------|----------|
| — | @pm/@dev/@designer | — | —h |

> 每个任务必须有交付物。"调研"不是任务，"调研报告"才是。

### Step 3：时间预排（5min）

| 时间段 | 做什么 |
|--------|--------|
| 周一-周二 | 核心开发/设计 |
| 周三 | 中期检查 + 调整 |
| 周四 | 完成 + 测试 |
| 周五上午 | 部署 + 推送用户 |
| 周五下午 | 复盘 |

> 周五必须留时间复盘。不复盘 = 白做。

## 周三：中期检查（10min）

回答 3 个问题：

1. **本周目标还在轨道上吗？** → 是/偏航/已废
2. **有什么阻碍需要解决？** → 具体阻碍 + 解决方案
3. **需要调整目标吗？** → 不变/缩小范围/换方向

> 偏航不是问题，不承认偏航才是问题。

## 周五：复盘（20min）

### Step 4：交付检查（5min）

| 交付物 | 完成度 | 备注 |
|--------|--------|------|
| — | ✅/⚠️/❌ | — |

### Step 5：假设验证（10min）

| 问题 | 回答 |
|------|------|
| 本周要验证的假设是什么？ | — |
| 数据/反馈怎么说？ | — |
| 假设成立吗？ | 成立/不成立/不确定 |

> 最重要的不是"做了多少"，而是"学到了什么"。

### Step 6：下周预规划（5min）

基于本周复盘，写下下周的方向（具体规划周一做）：

```
下周方向：[基于本周结论的下一步]
关键假设：[下周要验证什么]
```

## 产出

1. 周一规划（目标 + 任务拆解 + 时间预排）
2. 周三检查记录
3. 周五复盘报告（交付 + 假设验证 + 下周方向）

## 迭代节奏图

```
周一          周三          周五
┌──┐        ┌──┐        ┌──────┐
│规划│──────→│检查│──────→│复盘  │
│20m│       │10m│       │20m   │
└──┘        └──┘        └──────┘
  │                        │
  └── 本周做什么 ──────────┘── 本周学到了什么 → 下周做什么
```

## 反模式（避免）

- ❌ 一周定 5 个目标——你不是五个人
- ❌ 周五不复盘直接开始下周——不复盘 = 重复踩坑
- ❌ 周三发现偏航但假装没问题——早发现早调整
- ❌ 把"写了一堆代码"当成果——成果是"验证了假设"
- ❌ 规划 1 小时、执行 3 小时——规划不要超过 20 分钟
