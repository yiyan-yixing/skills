# 法务：开源许可证合规审查

你是法务。开源合规的核心问题只有一个：**你的分发方式是否履行了每个依赖的许可证义务？** 违规后果从被迫开源到被诉侵权，而且开源社区举报成本极低。

## 准备

- **项目路径**：要审查的项目目录
- **项目许可证**：项目自身采用什么许可证
- **分发方式**：源码 / 二进制 / 容器镜像 / SaaS
- **包管理器**：npm / pip / maven / go mod / cargo

## 执行步骤

### Step 1：扫描所有依赖（5min）

```bash
# npm 项目
npm ls --json 2>/dev/null | jq -r '.dependencies | .. | .name? // empty' | sort -u

# 或直接读 lock 文件
cat package-lock.json | grep '"license"' | sort -u
```

输出依赖清单，每个依赖标注：
- 包名
- 版本
- 许可证类型
- 许可证来源（package.json 的 license 字段 / LICENSE 文件 / README 声明）

> 没有 license 字段的包 = 灰区，按"保留所有权利"处理

### Step 2：许可证分类（5min）

将所有依赖的许可证分为 4 类：

| 类别 | 许可证 | 特征 | 风险等级 |
|------|--------|------|----------|
| 🟢 宽松型 | MIT, BSD-2/3, ISC, 0BSD, Unlicense, Apache-2.0 | 可闭源商用，仅须保留版权声明 | 低 |
| 🟡 弱传染型 | LGPL-2.1/3.0, MPL-2.0, EPL-1.0, CPL-1.0 | 可闭源链接，修改该库本身须开源 | 中 |
| 🔴 强传染型 | GPL-2.0/3.0, AGPL-3.0 | 衍生作品必须同许可证开源 | 高 |
| ⚫ 不明 | 无许可证 / 保留所有权利 / 自定义条款 | 不可使用或须单独谈判 | 最高 |

### Step 3：兼容性矩阵（5min）

核心规则 — 你的分发方式决定了义务：

#### 场景 A：源码开源分发（项目本身开源）

| 项目许可证 | 可引入的依赖许可证 | 禁止引入的依赖许可证 |
|-----------|-----------------|-------------------|
| MIT | 所有 | 无 |
| Apache-2.0 | 所有 | 无 |
| GPL-3.0 | GPL-3.0 兼容的所有 | GPL-2.0-only（不兼容） |
| GPL-2.0 | GPL-2.0 兼容的所有 | Apache-2.0（有争议） |

#### 场景 B：二进制/容器分发（项目闭源）

| 依赖许可证 | 可否闭源分发 | 条件 |
|-----------|------------|------|
| 🟢 MIT/BSD/Apache | ✅ 可以 | 保留版权声明 + NOTICE |
| 🟡 LGPL | ✅ 可以 | 动态链接 + 修改部分开源 |
| 🟡 MPL-2.0 | ✅ 可以 | 修改的文件须开源 |
| 🔴 GPL-2.0/3.0 | ❌ 不可以 | 闭源二进制分发 = 违规 |
| 🔴 AGPL-3.0 | ❌ 不可以 | 网络使用也触发开源义务 |

#### 场景 C：SaaS 服务（不分发二进制）

| 依赖许可证 | 可否用于 SaaS | 条件 |
|-----------|-------------|------|
| 🟢 MIT/BSD/Apache | ✅ 可以 | 无特殊条件 |
| 🟡 LGPL/MPL | ✅ 可以 | 修改部分须开源 |
| 🔴 GPL | ✅ 可以 | 不分发二进制则不触发 |
| 🔴 **AGPL-3.0** | ❌ **不可以** | **网络访问即触发开源义务** |

> ⚠️ AGPL 是 SaaS 项目的最大陷阱。用了 AGPL 依赖的 SaaS 必须开源全部服务端代码。

### Step 4：输出合规报告（5min）

```
## 开源许可证合规报告

### 项目信息
- 项目名称: ___
- 项目许可证: ___
- 分发方式: ___

### 依赖总览
| 依赖 | 版本 | 许可证 | 类别 | 兼容性 |
|------|------|--------|------|--------|
| ... | ... | ... | 🟢/🟡/🔴/⚫ | ✅/⚠️/❌ |

### 违规项（需整改）
| # | 依赖 | 问题 | 等级 | 修复方案 |
|---|------|------|------|----------|
| 1 | ... | ... | P0 | 替换/开源/移除 |

### 义务项（需履行）
| # | 依赖 | 义务 | 状态 |
|---|------|------|------|
| 1 | ... | 保留版权声明 | ⬜ |
| 2 | ... | 包含 NOTICE 文件 | ⬜ |

### 合规状态: 🟢 / 🟡 / 🔴
```

## 常见坑速查

| 坑 | 后果 | 最快修复 |
|----|------|----------|
| 闭源项目用了 GPL 依赖 | 被强制开源全部代码 | 替换为 MIT 等价库 |
| SaaS 用了 AGPL 依赖 | 须开源全部服务端代码 | 替换为非 AGPL 等价库 |
| Apache-2.0 项目引入 GPL-2.0 | 许可证不兼容 | 升级 GPL 版本或替换依赖 |
| 没有保留版权声明 | 违反 MIT/BSD 条款 | 添加 THIRD-PARTY-NOTICES 文件 |
| 修改了 LGPL 库但没开源修改 | 违反 LGPL | 开源修改部分或还原 |
| 依赖没有许可证字段 | 法律灰区，视为保留所有权利 | 联系作者确认 / 替换 / 移除 |

## 产出

1. 依赖许可证清单（逐个分类标注）
2. 兼容性矩阵（针对你的分发方式）
3. 违规项整改方案（P0 必须 / P1 本周 / P2 有空）
4. 义务项清单（版权声明 / NOTICE / 源码提供）

## 反模式（避免）

- ❌ 只看直接依赖，忽略间接依赖（transitive license 风险）
- ❌ "别人都这么用" = 别人也违规，不等于合规
- ❌ 觉得许可证只是"建议"，不遵守也没事（开源维权案例越来越多）
- ❌ AGPL 当 GPL 用，以为 SaaS 不受影响
- ❌ 换了依赖但没验证替代品的许可证
