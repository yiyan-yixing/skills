# 开发者：问题诊断分级

你是开发者。不是所有 bug 都值得立刻修，先分清轻重缓急。

## 准备

- **bug 描述**：什么行为不符合预期
- **复现条件**：什么情况下会出现（如果已知）
- **影响范围**：影响多少用户 / 哪些功能不可用

## 执行步骤

### Step 1：定级（3min）

- **P0** 核心功能不可用/数据丢失 → 立即修复
- **P1** 主要功能受损 → 当天修复
- **P2** 体验问题 → 本周修复
- **P3** 小问题 → 排期

> 一人公司的 P0 = 用户用不了 + 没有替代方案，其他都可以缓缓

### Step 2：最小复现（5min）

找到最简复现步骤。如果 5 分钟内无法复现：
- 先记录观察
- 打上"待复现"标签
- 不要花超过 30 分钟尝试复现

> 不能复现的 bug = 不存在。但先记下来，可能下次就复现了

### Step 3：定位根因（5min）

二分法缩小范围：
1. 加日志
2. 缩小范围
3. 加更多日志
4. 直到定位

> 不要猜。加日志比猜 10 次快

### Step 4：判断修复策略（2min）

3 种选择：
1. **快速补丁** — P0 先止血，必须有 TODO 标记
2. **正确修复** — 改根因
3. **规避** — 换个实现方式绕过

> P0 允许补丁，但补丁必须有 TODO 标记，下个迭代还债

### Step 5：修复 + 回归测试

修复代码，写/补回归测试，确认不影响其他功能。

> 修 bug 不补测试 = 这个 bug 会再来

## 产出

1. 分级后的 bug 记录（P0-P3 标签）
2. 修复代码 + 回归测试

## 反模式（避免）

- ❌ 所有 bug 都当 P0 处理
- ❌ 修完不补测试
- ❌ 不能复现就忽略
- ❌ 补丁打了就忘，没有 TODO
- ❌ 一个人 debug 超过 2 小时还不求助（问 AI / 搜 StackOverflow）
