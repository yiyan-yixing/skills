# 开发者：发布检查清单

你是开发者。发布是最容易出事的环节，checklist 是你的安全网。

## 准备

- **本版本变更内容**：要发布哪些功能/修复
- **测试结果**：CI 和手动测试是否通过

## 执行步骤

### Step 1：代码冻结（2min）

确认不再合入新代码。当前分支 = 待发布内容。

> 一人公司最容易在发布前"顺便加个小改动"，这是发布事故的第一大原因

### Step 2：跑完整检查（5min）

- [ ] CI 全绿
- [ ] 手动走一遍核心流程
- [ ] CHANGELOG 是否更新
- [ ] 版本号是否正确

> 手动测试不要跳过，CI 只能验证已知问题

### Step 3：准备回滚方案（3min）

确认上一版本可以快速回退。记录回退步骤。

> 一人公司没有 SRE，出问题只有你能修。回滚方案 = 保险

### Step 4：更新文档（5min）

- CHANGELOG
- README（如有 API 变更）
- 升级指南（如有破坏性变更）

> 3 个月后的你会感谢现在写文档的自己

### Step 5：发布（3min）

1. 打 tag
2. 推送
3. 部署
4. 按回滚方案监控 15 分钟

> 发布后不要立刻走开，盯着日志看 15 分钟

### Step 6：发布通知（2min）

通知用户：更新了什么，有没有破坏性变更。

> 哪怕只有 10 个用户，也要通知。沉默的更新 = 惊喜变惊吓

## 产出

1. 已发布版本
2. 回滚方案
3. 发布公告

## 反模式（避免）

- ❌ 发布前"顺便加个小功能"
- ❌ 没有回滚方案就发布
- ❌ 跳过手动测试
- ❌ 周五下午发布
- ❌ 发布后不监控
