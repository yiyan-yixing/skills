# 质疑协议

> 一人公司最大的风险是确认偏差——一个人既是作者又是审阅者。
> 质疑协议通过让不同 Agent 以对抗视角审查彼此的产出，弥补这个天然缺陷。

---

## 触发条件

| 事件 | 必须质疑 | 质疑者 | 质疑维度 |
|------|----------|--------|----------|
| @pm 提出 PRD | ✅ 必须 | @dev | 技术可行性：能不能做？多久？ |
| | ✅ 必须 | @data | 指标可量化：成功标准有没有数字？ |
| @dev 提出技术方案 | ✅ 必须 | @architect | 架构合理性：有没有更简单的方案？技术债多不多？ |
| @designer 出原型 | ⚠️ 建议 | @pm | 是否匹配需求：原型是否覆盖了核心场景？ |
| @ceo 做重大决策 | ✅ 必须 | @data | 数据支撑：有没有数据？数据够不够？ |
| @ops 提出增长方案 | ⚠️ 建议 | @data | 效果可测：能不能量化效果？ |

## 质疑流程

```
1. 协调者（主对话）识别触发条件
2. 协调者派质疑 Agent 独立审查
3. 质疑 Agent：
   a. 阅读被质疑的产出
   b. 以对抗视角找出问题和风险
   c. 写入 .claude/blackboard/challenges.md
4. 被质疑者回应：
   a. 阅读 challenges.md 中的质疑
   b. 修改产出 OR 给出合理理由保留原方案
   c. 在 challenges.md 中填写回应
5. 协调者判断：
   - 通过 → 继续推进
   - 修改 → 修改后重新质疑
   - 打回 → 推翻重来
```

## 质疑者行为准则

- **质疑不是否定** — 目的是让产出更好，不是证明对方错了
- **找风险不找茬** — 关注"可能出什么问题"，不关注"措辞好不好"
- **给出替代方案** — 说"这个有问题"必须同时说"你还可以这样做"
- **标注严重度** — 阻断/重要/建议，不要所有问题都标成阻断

## 质疑模板

质疑 Agent 的 system prompt 应包含：

```
你是质疑者。你的核心使命是以对抗视角审查 [对象]，找出可能的风险和问题。

审查维度：
1. [具体维度 1]
2. [具体维度 2]

审查方式：
- 独立思考，不要顺着原作者的思路走
- 想想什么情况下这个方案会失败
- 考虑边界和极端情况

输出格式（写入 challenges.md）：
| 编号 | 日期 | 质疑者 | 对象 | 问题 | 严重度 | 回应 | 结论 |
```

## 反模式

- ❌ 质疑者变成"橡皮图章"——走个形式就通过
- ❌ 被质疑者拒绝修改——"我觉得没问题"
- ❌ 所有质疑都标成"建议"——没有阻断级质疑说明质疑不够深入
- ❌ 质疑后不跟进——写了质疑但没人回应和判断
